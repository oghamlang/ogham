# Wire Formats & Serialization

## Overview

Ogham is a self-contained schema language. Code generators (`ogham-gen-*`) produce native language structs with built-in serialization. There is no dependency on `protoc` or `protoc-gen-*` in the primary pipeline.

## Architecture

```
*.ogham â†’ ogham compiler â†’ OghamCompileRequest â†’ ogham-gen-* plugins â†’ native code + serializers
```

- **One protocol**: all ogham plugins receive `OghamCompileRequest`. No `CodeGeneratorRequest`, no proto mode.
- **Self-contained output**: `ogham-gen-go` generates Go structs, `ogham-gen-rs` generates Rust structs, etc. Each with built-in serialization.
- **No protoc dependency**: the generated code does not depend on `protoc-gen-*` output.

## Supported Wire Formats

### Protobuf Wire (primary)

Every language plugin generates protobuf wire format serialization. This ensures binary compatibility with any protobuf implementation.

The generated code uses low-level protowire libraries (not protoc-gen-* generated code):

| Language | Protowire Library |
|----------|------------------|
| Go | `google.golang.org/protobuf/encoding/protowire` |
| Rust | `prost` or raw `bytes` encoding |
| TypeScript | `protobuf-es` encoding layer or custom |

Example (Go):

```go
// generated by ogham-gen-go
type User struct {
    Id        []byte
    FirstName string
    Email     string
}

func (u *User) MarshalProto() ([]byte, error) {
    var buf []byte
    buf = protowire.AppendBytes(buf, 1, u.Id)
    buf = protowire.AppendString(buf, 2, u.FirstName)
    buf = protowire.AppendString(buf, 3, u.Email)
    return buf, nil
}

func (u *User) UnmarshalProto(data []byte) error {
    for len(data) > 0 {
        num, typ, n := protowire.ConsumeTag(data)
        data = data[n:]
        switch num {
        case 1: u.Id, n = protowire.ConsumeBytes(data)
        case 2: u.FirstName, n = protowire.ConsumeString(data)
        case 3: u.Email, n = protowire.ConsumeString(data)
        default: n = protowire.ConsumeFieldValue(num, typ, data)
        }
        data = data[n:]
    }
    return nil
}
```

Wire-compatible with any `protoc-gen-*` generated code reading the same `.proto` schema.

### Proto JSON

Full compatibility with [Proto3 JSON Mapping](https://protobuf.dev/programming-guides/proto3/#json). This enables REST API interop without custom serialization logic.

Rules:
- Field names use `lowerCamelCase` by default (proto3 JSON convention)
- Enums serialize as string names
- `bytes` fields use base64 encoding
- `T?` fields are omitted when not set
- `Unspecified = 0` enum values are omitted
- Well-known types (`Timestamp`, `Duration`, etc.) use standard JSON representations

Each language plugin generates `MarshalJSON` / `UnmarshalJSON` (or equivalent) following these rules.

### Apache Arrow (planned)

Columnar serialization for analytics, data lake, and batch processing use cases. Ogham types map naturally to Arrow schemas:

| Ogham | Arrow |
|-------|-------|
| `bool` | `Boolean` |
| `string` | `Utf8` |
| `bytes` | `Binary` |
| `int32` | `Int32` |
| `int64` | `Int64` |
| `uint32` | `UInt32` |
| `uint64` | `UInt64` |
| `float` | `Float32` |
| `double` | `Float64` |
| `[]T` | `List<T>` |
| `T?` | nullable field |
| `map<K,V>` | `Map<K,V>` |
| `type` (message) | `Struct` |
| `enum` | `Dictionary<Int32, Utf8>` |

Arrow support enables direct export of Ogham-typed data to Parquet, IPC, and Flight without intermediate conversion.

## Proto Export

For users who need raw `.proto` files (for `protoc`, `buf`, gRPC toolchains, or third-party tools):

```bash
ogham proto export ./proto/
```

This generates standard `.proto` files from `.ogham` sources. Annotations are serialized as `OghamAnnotation` options (see language.md). The user then runs their own protoc pipeline independently.

Proto export is a **one-way export**, not part of the generation pipeline.

## Integration with protoc-gen-* Code

Ogham does not orchestrate `protoc-gen-*` plugins. However, language plugins **may** provide optional integration with protoc-generated code as a plugin-specific feature.

This is **not part of the Ogham specification** â€” it is a decision of each language plugin. For example, `ogham-gen-go` might offer an option to generate converters to/from `protoc-gen-go` structs. The mechanism (import paths, options) is plugin-specific because each language ecosystem has different layouts and conventions.

### Integration levels

A language plugin can support any combination:

| Level | Description | Example |
|-------|-------------|---------|
| **Standalone** | Own structs + protowire. No external dependencies. | `ogham-gen-go` default mode |
| **Proto struct converters** | Generate `ToProto()` / `FromProto()` for interop with protoc-gen-* types | `ogham-gen-go` with proto_package option |
| **Proto JSON** | JSON serialization compatible with proto3 JSON mapping | Built-in for all language plugins |
| **Arrow** | Columnar serialization for analytics | Planned |

## Summary

| Aspect | Decision |
|--------|----------|
| Plugin protocol | Always `OghamCompileRequest` |
| Primary wire format | Protobuf binary (via protowire libraries) |
| JSON format | Proto3 JSON Mapping compatible |
| Columnar format | Apache Arrow (planned) |
| protoc dependency | None â€” generated code is self-contained |
| Proto file export | `ogham proto export <dir>` â€” one-way, user manages protoc pipeline |
| protoc-gen-* integration | Language plugin decision, not part of spec |
